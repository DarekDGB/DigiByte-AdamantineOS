# Adaptive Core → AdamantineOS Governance Flow

## Purpose

This document describes the governance interaction between **Adaptive
Core v3** and **AdamantineOS**.

Adaptive Core can propose system upgrades through a deterministic
artifact called an **upgrade proposal**.\
AdamantineOS acts as the **verification boundary**, validating the
proposal, enforcing invariants, and producing a deterministic review
decision.

This architecture establishes a **trust-minimized governance handshake**
between the systems.

------------------------------------------------------------------------

# High Level Architecture

Adaptive Core: - Generates `upgrade_proposal_v3` artifact -
Canonicalizes proposal structure - Emits deterministic proposal hash

AdamantineOS: - Verifies artifact integrity - Recomputes canonical
proposal hash - Evaluates policy - Emits `proposal_review_receipt_v1`

------------------------------------------------------------------------

# Governance Flow

1.  Adaptive Core constructs proposal artifact.

2.  Proposal is canonicalized using deterministic JSON rules.

3.  Proposal hash is computed.

4.  Artifact is delivered to AdamantineOS.

5.  AdamantineOS verifies:

    -   schema compliance
    -   canonicalization
    -   proposal hash invariant

6.  AdamantineOS policy engine evaluates the proposal.

7.  AdamantineOS issues decision:

    -   APPROVE
    -   DENY

8.  A deterministic **review receipt** is produced.

------------------------------------------------------------------------

# Deterministic Invariants

The following invariants are enforced across repositories:

-   Canonical JSON serialization
-   Stable proposal hash
-   Artifact schema validation
-   Fail‑closed verification behavior

If any invariant fails, the proposal is rejected.

------------------------------------------------------------------------

# Trust Boundary

Adaptive Core is allowed to **propose**.

AdamantineOS is the authority that **verifies and decides**.

Adaptive Core cannot directly execute upgrades.

------------------------------------------------------------------------

# Security Model

The governance flow follows strict security principles:

-   Fail‑closed validation
-   Deterministic hashing
-   Canonical JSON normalization
-   Cross‑repository compatibility tests
-   No silent fallback behavior

------------------------------------------------------------------------

# Compatibility Lock (v2.1.0)

AdamantineOS v2.1.0 establishes compatibility with:

`upgrade_proposal_v3` artifacts generated by Adaptive Core v3.

Compatibility vectors are frozen in CI tests.

------------------------------------------------------------------------

# Design Goal

Create a verifiable governance handshake where:

**Proposals are created by Adaptive Core\
but trusted decisions are enforced by AdamantineOS.**
